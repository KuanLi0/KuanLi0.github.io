基础

risc-v的陷入机制

每一个risc-v寄存器都有一组控制寄存器，内核通过向这些寄存器写入内容来告诉CPU如何处理陷阱：

  • stvec：内核在在这里写入其陷阱处理的程序的地址；risc-v处理器跳转到这里执行。
  • sepc：发生陷阱时，risc-v会在这里保存pc，sret指令会将sepc复制到pc，进行返回
  • scause：描述陷阱原因
  • sscratch：内核在这里放置了一个值，这个值在陷阱处理程序一开始就会派上用场。
  • sstatus：其中的SIE位控制设备中断是否启用。如果内核清空SIE，RISC-V将推迟设备中断，直到内核重新设置SIE。SPP位指示陷阱是来自用户模式还是管理模式，并控制sret返回的模式。

上述寄存器用于在管理模式下处理陷阱，在用户模式下不能操作。

在陷入时，CPU不会切换内核页表，也不会切换内核栈，也不会保存除了pc之外的任何寄存器，内核软件必须执行这些任务。这样的目的是减少一些不必要的页表切换，提高灵活性。

从用户空间陷入

来自用户代码的陷阱比来自内核的陷阱更具挑战性，因为satp指向不映射内核的用户页表，栈指针可能包含无效甚至恶意的值。当发生trap时，首先会切换到uservec函数中。由于RISC-V硬件在陷阱期间不会切换页表，所以用户页表必须包括uservec（stvec寄存器指向的函数）的映射。uservec必须切换satp以指向内核页表；为了在切换后继续执行指令，uservec必须在内核页表中与用户页表中映射相同的地址。xv6使用包含uservec的蹦床页面（trampoline page）来满足这些约束。xv6将蹦床页面映射到内核页表和每个用户页表中相同的虚拟地址。这个虚拟地址是TRAMPOLINE。

蹦床内容在***trampoline.S***中设置，并且（当执行用户代码时）stvec设置为uservec 。uservec负责寄存器的保存。但是uservec需要能够修改一些寄存器，以便设置satp并生成保存寄存器的地址。RISC-V以sscratch寄存器的形式提供了帮助。uservec开始时的csrrw指令交换了a0和sscratch的内容。现在用户代码的a0被保存了；uservec有一个寄存器（a0）可以使用；a0包含内核以前放在sscratch中的值。因为satp仍然指向用户页表，所以uservec需要将陷阱帧映射到用户地址空间中。每当创建一个进程时，xv6就为该进程的陷阱帧分配一个页面，并安排它始终映射在用户虚拟地址TRAPFRAME，该地址就在TRAMPOLINE下面。尽管使用物理地址，该进程的p->trapframe仍指向陷阱帧，这样内核就可以通过内核页表使用它。

陷阱帧包含指向当前进程内核栈的指针、当前CPU的hartid、usertrap的地址和内核页表的地址。uservec取得这些值，将satp切换到内核页表，并调用usertrap。

usertrap的任务是确定陷阱的原因，处理并返回(*kernel/trap.c*:37)。如上所述，它首先改变stvec，这样内核中的陷阱将由kernelvec处理。它保存了sepc（保存的用户程序计数器），再次保存是因为usertrap中可能有一个进程切换，可能导致sepc被覆盖。如果陷阱来自系统调用，syscall会处理它；如果是设备中断，devintr会处理；否则它是一个异常，内核会杀死错误进程。系统调用路径在保存的用户程序计数器pc上加4，因为在系统调用的情况下，RISC-V会留下指向ecall指令的程序指针（返回后需要执行ecall之后的下一条指令）。在退出的过程中，usertrap检查进程是已经被杀死还是应该让出CPU（如果这个陷阱是计时器中断）。

返回用户空间的第一步是调用usertrapret (*kernel/trap.c*:90)。该函数设置RISC-V控制寄存器，为将来来自用户空间的陷阱做准备。这涉及到将stvec更改为指向uservec，准备uservec所依赖的陷阱帧字段，并将sepc设置为之前保存的用户程序计数器。最后，usertrapret在用户和内核页表中都映射的蹦床页面上调用userret；原因是userret中的汇编代码会切换页表。

Backtrace

该实验是实现一个打印曾经调用过的函数地址，比较简单。

void printbt() {
  uint64 fp = r_fp();
  printf("backtrace:\n");
  while (PGROUNDDOWN(fp) != PGROUNDUP(fp))
  {
    /* code */
    uint64 ra = *(uint64*)(fp - 8); //return address
    printf("%p\n", ra);
    fp = *(uint64*)(fp - 16);
  }
  
}

该函数主要使用PGROUNDDOWN和PGROUNDUP这两个宏来，当前的fp是否在跃出边界。返回地址存在fp（栈帧的基址）寄存器中-8的位置，上一个栈帧的地址存在fp-16的位置。

Alarm

这项练习要实现定期的警报,用来模拟用户级的异常处理。在给的测试用例中（user/alarmtest.c），会调用sigalarm(2, periodic)函数，使内核每2个ticks就调用periodic函数，该函数会打印“alarm！”字符。通过修改内核，使得内核可以调转到位于用户空间的处理函数即periodic，他将打印“alarm！”字符。

根据xv6的trap原理，不加alarm的流程为（以系统调用为例）：用户程序会跳转到trapline中的uservec函数，这里会将pc值保存到sepc寄存器，并将上下文保存到trapframe，然后跳转到usertrap函数中，在该函数中，会将sepc的值保存到p->trapframe->epc的数据结构中，并加4。然后usertrap根据trap类型去执行不同的函数。执行完usertrap函数最后会调用usertrapret函数，在该函数中，会取保存在当前进程的trapframe中的epc写入到pc寄存器中。最后跳转到userret函数中，恢复寄存器，在通过sret指令返回到ecall的下一条指令。

但是，如果要加alarm功能，需要对上面的步骤进行修改，具体如下：

增加进程的成员变量

在proc结构体中，增加成员变量，并且对这些字段进行初始化和释放

  int alarm_intervel;          //alarm cycle, 0 represents prohibit alarm.
  void(*lk_alarm_handle)();    //alarm callback handle function.
  int lk_alarm_ticks;          //current alarm signals nums.
  struct trapframe* lk_alarm_trapframe; //process's trap frame when alarm interrupt.
  int lk_alarm_goingoff;       //Is there already a clock interrupt executing and not yet returned.

[!NOTE]

按照常规来说，只需要添加报警间隔和报警处理函数以及两次报警间的计数即可。但是考虑到handle函数可能会更改用户寄存器，所以需要一个陷阱帧进行提前保存，并且还要防止在alarm_handle函数中再次调用alarm_handle函数，导致陷阱帧被覆盖，所以需要设置一个变量标记是否已经有一个时钟中断正在运行。

实现sys_sigalarm和sys_sigreturn系统调用

sys_sigalarm主要获取参数，并调用处理函数。

//kernel/sysproc.c
uint64 sys_sigalarm(void) {
  int n;
  uint64 fn;
  if (argint(0, &n) < 0)
    return -1;
  if (argaddr(1, &fn) < 0)
    return -1;

  return lk_sigalarm(n, (void(*) ()) (fn));
}

uint64 sys_sigreturn(void) {
  return lk_sigreturn();
}

[!NOTE]

sys_sigalarm和lk_sigalarm函数只是一层包装，真正的实现在下面两个函数。

在trap.c中实现具体的代码：

//set the clock related properties in the process
int lk_sigalarm(int ticks, void(*handle)()) {
  struct proc* p = myproc();
  p->alarm_intervel = ticks;
  p->lk_alarm_handle = handle;
  p->lk_alarm_ticks = ticks;
  return 0;
}

//restore the process to the state before the alarm interrupt 
int lk_sigreturn() {
  struct proc* p = myproc();
  *p->trapframe = *p->lk_alarm_trapframe;
  p->lk_alarm_goingoff = 0;
  return 0;
}

[!NOTE]

注意：在lk_sigalarm中设置ticks以及中断处理函数的地址。在lk_sigreturn函数中，恢复陷阱帧。

在usertrap中实现中断触发的代码

在usertrap中实现中断触发的代码，主要将pc设置为用户自定义的处理函数

 // give up the CPU if this is a timer interrupt.
  if(which_dev == 2) {
    if(p->alarm_intervel != 0 && --p->lk_alarm_ticks <= 0 && p->lk_alarm_goingoff == 0) {
      //make sure the click is set && the click countdown ends && no other clocks are running

      p->lk_alarm_ticks = p->alarm_intervel; //reset the clock countdwon 
      *p->lk_alarm_trapframe = *p->trapframe; //save current trapframe
      p->trapframe->epc = (uint64)p->lk_alarm_handle; //jump clock callback function
      p->lk_alarm_goingoff = 1; //mark that there is a clock running
    }
    yield();
  }

该实验的函数调用流程大概是sys_sigalarm->lk_sigalarm(设置参数)->usertrap(保存陷阱帧)->handle（用户自定义函数）-> lk_sigreturn(恢复陷阱帧)。
